# 场景：技术选型 / 技术评估 (Technical Evaluation)

## 何时触发
用户问："X 和 Y 用哪个"、"X 技术栈选型"、"X 框架对比"、"X benchmark"、"X 性能怎么样"、"我们该不该用 X"、"X 的技术债"。

> 这是生态空白点 —— 现有 skill 都把技术评估塞进通用 research，没有专门的 SOP。本场景填补这个空白。

## 必答 6 大问题

1. **功能匹配度**：候选方案的功能 vs 用户需求的 fit 度
2. **性能 benchmark**：实测数据（不是 marketing 数据）
3. **集成复杂度**：上手难度 / 学习曲线 / 团队适配
4. **TCO（总拥有成本）**：license + 基础设施 + 人力 + 迁移
5. **风险评估**：技术风险 + 供应商风险 + 锁定风险
6. **决策建议**：基于场景的明确选择 + 反方反驳

## 推荐方法论

### 1. 候选方案筛选（先做减法）
- **初筛 maturity gate**：剔除 star <500（开源）或 没有付费客户案例（商业）的方案
- **二筛 use-case fit**：用户的核心需求是什么？候选方案的"招牌能力"是否对得上？
- **三筛 ecosystem**：社区活跃度 / 文档完整度 / 第三方集成 / 招聘市场
- 最终选 3-5 个候选深入对比

### 2. 功能矩阵（必须）
建一个表：`feature × candidate`
- 行：用户需求里的所有功能点（按重要性排序：Must / Should / Nice）
- 列：候选方案
- 单元格：是 / 否 / 部分支持 + 备注（如版本要求、配置成本）
- 必须区分"官方支持" / "插件支持" / "需要自己写"

### 3. Benchmark 实测原则
- **不信 marketing benchmark**（厂商自测的数据）
- 找第三方 benchmark：MLPerf / TPC / 独立技术博客 / 大规模生产环境的复盘
- 关键指标按场景：吞吐 / 延迟 P50 P95 P99 / 准确性 / 稳定性 / 弹性
- 标注 benchmark 的硬件 / 数据规模 / 测试方法 / 测试时间

### 4. TCO 拆解（5 项必算）
| 成本项 | 含义 |
|--------|------|
| License/订阅 | 商业产品的 per-seat / per-usage 费用 |
| 基础设施 | 服务器 / GPU / 带宽 / 存储 / DB |
| 人力 | 学习 / 实施 / 维护工程师人数 × 时间 |
| 迁移 | 数据迁移 / 业务停机 / 培训 / 风险缓冲 |
| 锁定 | switch cost 估算（如果未来要换） |

按 3 年总成本算，不要只看年度。

### 5. 风险评估（4 类）
- **技术风险**：方案是否成熟？bug 多不多？是否有重大架构缺陷？
- **供应商风险**：公司财务状况 / 是否可能被收购 / 主要 contributor 是否还在
- **锁定风险**：数据格式 / API / 团队技能的可迁移性
- **合规风险**：开源 license（GPL/AGPL 传染性？）/ 数据驻留 / 行业合规

### 6. 决策建议（必须明确）
不允许"看情况"。报告结尾必须有：
- **如果你的场景是 A** → 选 X
- **如果你的场景是 B** → 选 Y
- **以下情况建议放弃所有候选** → 列出几个 dealbreaker

并附上反方质疑："但如果 Z 发生，这个推荐可能错，需要重新评估"

## 推荐信源
- **GitHub**：star 数 / commit 频率 / issue 响应 / contributor 多样性
- **Papers with Code**：算法类 SOTA 跑分
- **Stack Overflow / Reddit / HackerNews**：实际使用反馈
- **InfoQ / The Information / 各家 engineering blog**：生产实践 case study
- 厂商官方文档（但只信功能描述，不信 benchmark）
- 中文：极客时间 / InfoQ 中国 / 阿里/腾讯/字节技术博客
- 必须含 ≥3 个真实生产案例（不是 marketing demo）

## 推荐模板
`templates/report.md` + 内嵌功能矩阵 + TCO 表

## 输出 deliverables
1. **TL;DR**（一句话推荐 + 主要 trade-off）
2. **候选方案介绍**（每个 ≤200 字）
3. **功能矩阵**（必须）
4. **Benchmark 对比**（含数据来源 + 测试条件）
5. **TCO 对比**（3 年总成本）
6. **风险评估**（4 类各打分）
7. **决策建议**（分场景的明确推荐）
8. **反方质疑 + 监控指标**（如果选了 X，未来出现什么信号要重新评估）

## 反模式
- 信厂商 marketing benchmark → 必须找第三方
- 只看 star 数 → 必须看 commit 频率、issue 响应、文档质量
- "X 和 Y 都很好，看你的需求" → 禁止，必须给明确推荐
- 没算 3 年 TCO → 必须算
- 没考虑团队当前技能栈 → 必须把"团队当前会用 X" 当作一个评估维度
- 没有反方质疑 → 必须有
